# Smart Logistics

Node.js/TypeScript warehouse inventory system with a REST API, JWT authentication, a modern landing page, and a protected operations dashboard.

## Stack

- Node.js 20+
- TypeScript
- Express
- React.js + Vite frontend
- Tailwind CSS
- PostgreSQL
- JWT authentication
- Zod validation
- Vitest + pg-mem tests
- Swagger UI / OpenAPI

## Setup

```bash
npm install
cp .env.example .env
# Edit DATABASE_URL, JWT_SECRET, AUTH_USERNAME, and AUTH_PASSWORD in .env
npm run migrate
npm run seed
npm run dev
```

The app starts on `http://localhost:3000` by default for local development.

Frontend routes and public assets are served by Express from `public/`:

- Landing page: `http://localhost:3000/`
- Dashboard: open from the landing page after sign-in
- Swagger UI: `http://localhost:3000/docs`
- OpenAPI JSON: `http://localhost:3000/openapi.json`

Optional seed data is available in `seed.sql`. Run `npm run seed` after migrations to create mixed standard/cold warehouses, stockable SKUs, low-stock examples, and deterministic inventory-transfer scenarios for success, capacity failure, and storage-type failure checks.

Local URLs:

- Landing page: `http://localhost:3000/`
- Health: `GET http://localhost:3000/health`
- Swagger UI: `http://localhost:3000/docs`
- OpenAPI JSON: `GET http://localhost:3000/openapi.json`

The repository also includes a static OpenAPI submission artifact at `docs/openapi.json`.

Protected endpoints require `Authorization: Bearer <access-token>`. Public endpoints are `/health`, `/docs`, `/openapi.json`, and `/auth/token`.

## Frontend

The frontend is a React.js corporate dashboard experience built with Vite and Tailwind CSS:

- `client/src/App.tsx` owns the app shell, auth state, React Router setup, session token, and dashboard data refresh flow.
- `client/src/pages/` contains route-level screens for the landing page and dashboard.
- `client/src/components/` contains reusable UI primitives such as buttons, brand, login dialog, header, and metric cards.
- `client/src/lib/` contains API, normalization, session token, and metric helpers.
- `client/src/types/` contains shared frontend TypeScript types.
- `client/src/styles.css` loads Tailwind layers and base styling.
- `client/public/favicon.svg` provides the Smart Logistics favicon source.
- `public/` contains the compiled frontend bundle generated by `npm run build:client` and served by Express.

Useful frontend commands:

```bash
npm run dev:client    # Vite dev server on http://localhost:5173 with API proxying
npm run build:client  # Type-check and compile React/Tailwind assets into public/
npm run build         # Build the frontend and TypeScript API server
```

The login form posts to `/auth/token` using `AUTH_USERNAME` and `AUTH_PASSWORD`. On success, the returned JWT is stored in session storage under `smart.accessToken`, then used as a Bearer token for `/warehouses` and `/inventory/report`.

If the user is not signed in, opening the dashboard presents the auth dialog. After sign-in, the dashboard loads live API data where available and exposes refresh-safe routes such as `/dashboard/overview`, `/dashboard/warehouses`, `/dashboard/items`, and `/dashboard/inventory`.

Frontend additions and UX behavior:

- Route-based dashboard navigation with refresh persistence.
- Toast-style success/error feedback. Error toasts use a red background.
- CRUD modals for warehouses, items, and inventory actions.
- Confirmation dialogs for destructive deletes.
- Warehouse and item tables with pagination, row action menus, view dialogs, and search-aware data.
- Inventory report table showing total capacity, current occupancy, percent full, stored items, and low-stock counts.
- Compact stored-item summaries instead of rendering every SKU as a large badge.
- Overview inventory health section with most-stocked and least-stocked items.
- Low-stock flags when a warehouse item quantity is below 10.
- Header search scoped to the current page:
  - Overview search shows matching dashboard page links.
  - Warehouses search filters warehouse rows.
  - Items search filters SKU rows.
  - Inventory search filters report rows and stored items.

Frontend validation mirrors backend rules for better UX while keeping the API as the final source of truth:

- SKU format validates as `ABC-12345-X`.
- Negative quantities are rejected.
- Add/transfer quantity must be greater than 0.
- Set quantity can be 0 or greater.
- Transfer source and destination warehouses must be different.
- Inventory forms require an existing SKU from the item catalog.
- Cold-storage SKUs are blocked from standard warehouses before submit.
- Obvious capacity overfill is blocked before submit.
- Non-empty warehouse deletes are blocked in the UI before the API request.

Required environment variables:

- `DATABASE_URL`
- `DATABASE_SSL` optional, defaults to `false` for local PostgreSQL
- `JWT_SECRET`
- `AUTH_USERNAME`
- `AUTH_PASSWORD`
- `PORT` optional, defaults to `3000`
- `JWT_EXPIRES_IN` optional, defaults to `1h`

Local `.env` example:

```env
DATABASE_URL=postgres://postgres:postgres@localhost:5432/smart_logistics
DATABASE_SSL=false
PORT=3000
NODE_ENV=development
JWT_SECRET=replace-with-a-long-random-secret
JWT_EXPIRES_IN=1h
AUTH_USERNAME=admin
AUTH_PASSWORD=change-me
```

## Tests

```bash
npm test
```

The test suite covers transfer success, insufficient destination capacity, incompatible cold/standard storage, insufficient source stock, validation failures, capacity failure on add, zero-value set inventory, and cold item add prevention.

## Database Model

- `warehouses`: name, location, `max_capacity`, type (`STANDARD` or `COLD`), and `deleted_at` for soft deletes.
- `items`: name, unique SKU, storage requirement (`STANDARD` or `COLD`), and `deleted_at`.
- `inventory`: join table tracking quantity per `(warehouse_id, item_id)`.

SKU format is enforced at both API and database level: `^[A-Z]{3}-\d{5}-[A-Z]$`.

## Business Rules

- Warehouse, item, and inventory endpoints are protected by JWT auth.
- Cold items cannot be added or transferred into standard warehouses.
- Adds/transfers fail if the destination warehouse capacity would be exceeded.
- Inventory mutations use explicit warehouse row locks with `SELECT ... FOR UPDATE`.
- Capacity-sensitive inventory routes run under `READ COMMITTED` with warehouse locks, so concurrent requests for the last available slots are serialized: the second request waits, then re-reads occupancy after the first commit.
- Transfers lock source/destination warehouse rows in stable sorted order to reduce deadlock risk and prevent race-condition overfills.
- Transfers move inventory by item resolved from the submitted SKU, preventing ghost stock or cross-SKU movement.
- Warehouses can only be soft-deleted when empty.
- Reports are paginated with `warehousePage` and `warehousesPerPage` for warehouses, plus `warehouseItemPage` and `warehouseItemsPerPage` for each warehouse item list. Both page-size fields max out at 100.

## Seed Data

`npm run seed` applies `seed.sql`, which includes:

- Mixed `STANDARD` and `COLD` warehouses.
- Standard and cold SKUs.
- Inventory rows with quantities below 10 to demonstrate `Low Stock`.
- Transfer test warehouses:
  - `Transfer Source Standard`
  - `Transfer Destination Standard`
  - `Transfer Source Cold`
  - `Transfer Blocked Standard`
- Transfer test SKUs:
  - `TST-90001-A`
  - `TCL-90002-C`
  - `LOW-90003-A`

The seed file also removes invalid cold-stock rows from standard warehouses before inserting deterministic stock rows.

## Error Format

All errors use the same shape:

```json
{
  "error": "INSUFFICIENT_CAPACITY",
  "message": "Destination warehouse does not have enough remaining capacity.",
  "code": 422
}
```

## API Examples

Get a JWT:

```bash
curl -X POST http://localhost:3000/auth/token \
  -H "Content-Type: application/json" \
  -d '{"username":"admin","password":"change-me"}'
```

Use the returned `accessToken` as a Bearer token for warehouse, item, and inventory endpoints.

Create a warehouse:

```bash
curl -X POST http://localhost:3000/warehouses \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer <access-token>" \
  -d '{"name":"Cold Manila","location":"Manila","maxCapacity":100,"type":"COLD"}'
```

Create an item:

```bash
curl -X POST http://localhost:3000/items \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer <access-token>" \
  -d '{"name":"Ice Cream","sku":"ICE-12345-C","storageRequirement":"COLD"}'
```

Add inventory:

```bash
curl -X POST http://localhost:3000/inventory/add \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer <access-token>" \
  -d '{"warehouseId":"<warehouse-id>","sku":"ICE-12345-C","quantity":25}'
```

Transfer inventory:

```bash
curl -X POST http://localhost:3000/inventory/transfer \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer <access-token>" \
  -d '{"fromWarehouseId":"<source-id>","toWarehouseId":"<destination-id>","sku":"ICE-12345-C","quantity":5}'
```

Inventory report:

```bash
curl "http://localhost:3000/inventory/report?warehousePage=1&warehousesPerPage=50&warehouseItemPage=1&warehouseItemsPerPage=50" \
  -H "Authorization: Bearer <access-token>"
```

Delete an empty warehouse:

```bash
curl -X DELETE http://localhost:3000/warehouses/<warehouse-id> \
  -H "Authorization: Bearer <access-token>"
```

## Local Run Notes

1. Start local PostgreSQL.
2. Create a database matching `DATABASE_URL`, for example `smart_logistics`.
3. Copy `.env.example` to `.env` and adjust credentials if needed.
4. Run migrations and seed data.
5. Start the Express app.

```bash
createdb smart_logistics
npm run migrate
npm run seed
npm run dev
```
